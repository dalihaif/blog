## 📖 从一个悲惨的故事说起

小李是医院办公室的文员。有一天，领导让他写一份材料。

他打开电脑，开始写——

```
汇报材料.doc
汇报材料_修改版.doc
汇报材料_最终版.doc
汇报材料_最终版2.doc
汇报材料_最终版_领导改.doc
汇报材料_打死不改版.doc
汇报材料_打死不改版2.doc
汇报材料_真的最终版.doc
汇报材料_定稿_张院改.doc
汇报材料_定稿_李院又改.doc
汇报材料__绝对不改了__.doc
```

最后，他自己都搞不清哪个才是最新的。更要命的是，他不小心覆盖了一个重要版本，领导要之前那段话，找不回来了。

小李哭了。

**如果小李用了 Git，这一切都不会发生。**

## 🤔 Git 到底是个啥？

一句话：**Git 是一个「时光机」，能帮你记住文件的每一次修改，随时回到过去。**

你不需要懂代码才能用 Git。它就像一个超强的「撤销」功能——不只是撤销一步，而是能撤销到任意一步，而且永远不会丢失。

打个比方：

| 没有 Git | 有 Git |
|----------|--------|
| 写论文只有一份文件，改错了就完了 | 每次修改都自动存档，想回到哪版回到哪版 |
| 文件名叫「最终版」「最终版2」「打死不改版」 | 所有版本都在，但文件夹里永远只有一个文件 |
| 改坏了只能从头重写 | 一条命令回到任意历史版本 |
| 两个人改同一份文件，合并时手动对 | Git 自动帮你合并，冲突了会提示你 |

## 🎬 没有 Git vs 有 Git：三个真实场景

### 场景一：写材料

**没有 Git 的人生：**

你写了一下午的材料，快写完了，领导说"第一段那个意思不对，改一下"。你改了，写完发现还是之前的好。但是之前的已经被覆盖了，你只能凭记忆重新写一遍。

**有 Git 的人生：**

你写完材料，跟 Git 说"帮我记住现在的样子"（这叫**提交**）。领导让你改，你改完又说"帮我记住"。后来你想回到之前的版本？一条命令，瞬间穿越回去。

就像游戏里的**存档**——每到一个关卡就存一次，打不过可以读档重来。

### 场景二：多人协作

**没有 Git 的人生：**

张三写第一章，李四写第二章，你负责合并。张三发来一个 Word，李四发来一个 Word，你手动把两份拼在一起。结果格式全乱了，你花了一整天调格式。第二天张三说"我那段又改了"，你崩溃了。

**有 Git 的人生：**

张三改他的，李四改他的，Git 自动记录每个人的修改。你点一下"合并"，Git 把两个人的修改自动合到一起。如果有冲突（两个人改了同一行），Git 会标出来让你选——**不会偷偷覆盖任何人的劳动成果**。

### 场景三：改出 bug

**没有 Git 的人生：**

你的网站本来好好的，今天加了个新功能，结果整个网站崩了。你记得是改了某个文件导致的，但改了哪里？不记得了。于是你盯着代码一行行找，找了两个小时，头发掉了一把。

**有 Git 的人生：**

Git 清清楚楚记录了你改了哪个文件、改了哪一行、什么时间改的。你甚至可以让 Git 告诉你"今天和昨天相比，我改了什么"，一眼就能定位问题。

## 🏗️ Git 的核心概念（只有4个）

别被术语吓到，我把 Git 的核心概念翻译成人话：

### 1. 仓库（Repository）—— 一个项目的大文件夹

仓库就是你的项目文件夹，只不过它多了一个隐藏的 `.git` 目录，里面存着所有的历史记录。

你可以理解为一个**魔法文件夹**——表面上跟普通文件夹一样，但暗地里它记住了你在这个文件夹里做过的每一件事。

### 2. 提交（Commit）—— 存一次档

每次你觉得"现在的状态不错，帮我记住"，就做一个**提交**。

提交就像游戏存档——你给它起个名字（比如"完成了第一章"），Git 就把当前所有文件的状态保存下来。

以后你可以随时回到这个存档点。

### 3. 分支（Branch）—— 平行宇宙

你正在写材料，突然领导说"先出一版应急的，内容少一点"。

没有分支：你把当前文件复制一份，改出应急版，改完再手动把修改合并回正式版。

有分支：你开一条新分支（相当于平行宇宙），在分支里随便改，改好了合并回来。**主线上的一切完全不受影响**。

就像你在考试时先在草稿纸上算，算对了再抄到卷子上。分支就是草稿纸。

### 4. 远程仓库（Remote）—— 云备份

你的代码存在自己电脑上，万一电脑坏了怎么办？

**远程仓库**就是把你的仓库再存一份到云端（比如 GitHub）。这样就算你电脑掉沟里了，代码还在云端好好的。

而且，别人也能从远程仓库拿到你的代码，实现多人协作。

## 🎮 五个命令走天下

Git 有很多命令，但日常用的就这五个：

### 1. `git init` —— 开启魔法

在一个普通文件夹里输入这个命令，它就变成了 Git 仓库，开始记录你的一举一动。

```
你：git init
Git：好的，从现在起，这个文件夹里的所有变化我都帮你记着。
```

> 💡 一个项目只需要执行一次。

### 2. `git add` —— 选出要存档的文件

你改了三个文件，但只想存其中两个？用 `git add` 选出来。

```
git add 汇报材料.doc     # 存这一个
git add .                 # 存所有修改过的文件（最常用）
```

> 💡 可以理解为"把要存档的东西放进篮子里"。

### 3. `git commit` —— 存档！

选好文件后，正式存档：

```
git commit -m "完成了第一版材料"
```

`-m` 后面是存档说明，**一定要写清楚**，不然以后你看着一堆"更新""修改""fix"的说明，完全不知道哪个是哪个。

> 💡 就像给游戏存档起名字——"打败Boss前""拿到宝剑后"，一看就知道。

### 4. `git push` —— 上传到云端

本地存好档了，推到远程仓库备份一下：

```
git push origin master
```

> 💡 相当于把你的游戏存档同步到云盘，手机上也能接着玩。

### 5. `git pull` —— 从云端拉取最新内容

别人改了代码推到了云端，你本地还是旧的？拉一下就同步了：

```
git pull origin master
```

> 💡 相当于打开云盘，下载最新的存档。

## 🔄 日常工作流：就这么简单

实际工作中，你每天的流程大概是：

```
早上开始工作
    ↓
git pull            ← 先拉一下别人的最新修改
    ↓
写代码 / 改文件 / 做东西
    ↓
git add .           ← 把修改放进篮子
    ↓
git commit -m "今天干了什么"   ← 存档
    ↓
继续干活……
    ↓
下班前
    ↓
git push            ← 推到云端，回家！
```

就这么简单。**拉代码 → 干活 → 存档 → 推代码**，四步走天下。

## 🆘 常用急救操作

### "我改错了，想回到上一个版本"

```
git log              ← 看看历史存档，找到想回到的那个版本号
git checkout 版本号   ← 穿越回去
```

### "我改了一半，还没存档，但想暂时放下"

```
git stash            ← 把当前修改先藏起来
git stash pop        ← 之后取出来继续干
```

> 就像你在写作业，突然要去吃饭，先把作业本合上放一边，吃完饭打开继续写。

### "我想看看我改了什么"

```
git diff             ← 看看当前修改和上次存档的区别
git status           ← 看看哪些文件被改了
```

### "我不小心提交了，想撤回"

```
git reset --soft HEAD~1    ← 撤回上一次提交，修改还在
```

> 注意：`--soft` 是温柔的撤回（修改还留着），`--hard` 是暴力的撤回（修改也没了）。一般用 `--soft` 就行。

## 🛠️ Windows 上装 Git

说了这么多，还没说怎么装。很简单：

1. 打开 https://git-scm.com/download/win
2. 下载安装包，双击安装，**一路默认下一步**
3. 装完后右键菜单会多出 **"Git Bash Here"**，点它就能用

> 💡 推荐搭配 [GitHub Desktop](https://desktop.github.com/)，图形界面操作，不用敲命令，点按钮就行。

## 💡 写在最后

Git 最初是 Linux 之父 Linus Torvalds 给自己管理代码用的。他觉得市面上的版本管理工具都太难用了，干脆自己写一个。

结果这个工具现在全世界程序员都在用。

但 Git 不只是程序员的工具。**任何需要反复修改、多人协作的工作，都能用 Git 来管理**——写论文、写方案、做设计、甚至写小说。

核心思想就一个：**别怕改，改了都能回来。**

有了这个底气，你才能真正放开手脚去创作。
