## 📖 写在前面

在医院上班的人都有这个体验——想打个电话找人，先翻厚厚的纸质通讯录，找到号码再手动拨。号码如果更新了，那本册子还得重新印。

我们医院有 2000 多人，加上各科室的办公电话，总共 2648 条记录。之前大家查号码的方式大致分三种：

- 找档案室要最新的 Excel 文件
- 微信通讯录里翻人名
- 直接打总机让接线员帮忙转

第一种麻烦，第二种不全，第三种占线。我就想，能不能做一个**装在手机里的通讯录**，打开就能搜，点一下就拨号，不用装 App，不用上应用商店。

答案当然是能。而且用现在的技术，一个人花几天就能做出来。

---

## 🎯 要解决什么问题？

先列一下真实场景里的痛点：

| 场景 | 现在的做法 | 问题 |
|------|------------|------|
| 找某个医生电话 | 翻纸质通讯录 | 册子不在手边，号码过期 |
| 找某个科室分机 | 打总机转 | 总机忙，等半天 |
| 新人入职 | 问同事要通讯录 | 拿到的是三个月前的旧版 |
| 下乡镇义诊 | 现场没有网络 | 纸质通讯录带少了，查不到 |
| 想存到手机 | 一条条手动录 | 2000多人，录到明年 |

一句话：**号码数据有，但获取方式太落后**。

---

## 💡 为什么做成 PWA？

最开始有人建议我做成一个安卓 App，上架应用商店。我算了算成本：

- 开发要学 Kotlin 或 Flutter，学习周期长
- 上架要审核，安卓市场还多，每个都要单独弄
- 更新要发新版本，用户要手动更新
- 苹果手机还要单独做 iOS 版

算了吧。

**PWA（Progressive Web App，渐进式网页应用）** 刚好解决这些问题：

- 不用上架，浏览器打开就能用
- 手机浏览器菜单里点"添加到主屏幕"，桌面上就出现一个图标，跟原生 App 一模一样
- 打开后自动缓存数据，下次没网也能查
- 更新只要在服务器上替换文件，用户下次打开自动生效

对医院内部工具来说，这是最合适的技术路线。

## 🔧 核心功能

### 1. 智能搜索——支持拼音首字母

这是使用频率最高的功能。搜"李海峰"能找到，搜"lhj"也应该能找到。

实现方式是在数据里给每个人名预计算了拼音首字母，搜索时同时匹配**中文姓名、科室名、拼音首字母**，输入"bys"能自动匹配到"病案室"。

```javascript
// 搜索逻辑核心：同时匹配中文和拼音首字母
function matchRecord(record, keyword) {
  const k = keyword.toLowerCase();
  return (
    record.name.includes(k) ||           // 中文姓名
    record.dept.includes(k) ||            // 科室名
    record.pinyin.includes(k) ||          // 拼音首字母（如 lhf）
    record.phone.includes(k)              // 电话号码
  );
}
```

搜索结果实时刷新，输入瞬间就能看到结果，不需要点"搜索"按钮。

### 2. 一键拨号

在手机上打开通讯录，点一下电话号码，自动跳出拨号界面，再点一下就拨出去了。

办公电话还做了**区号自动补全**——数据里存的是"2201062"，显示时自动变成"0872-2201062"，点拨号时直接拨完整号码，不用手动加区号。

### 3. 安装到桌面

这是 PWA 的杀手功能。步骤很简单：

1. 手机浏览器打开通讯录地址
2. 点浏览器菜单 → **"添加到主屏幕"**
3. 桌面上出现「通讯录」图标
4. 以后点这个图标就直接打开，跟打开微信一样快

第一次打开时会把所有数据缓存到手机上，以后**没网也能查**。

### 4. Excel 导入导出

通讯录数据来源是一个 Excel 文件（`通讯录_20260519(4).xlsx`），我先把它转成了 JSON 格式存在服务器上。

但数据会更新，所以系统支持：

- **导入**：管理员上传新的 Excel 文件，自动解析并替换旧数据
- **导出**：把当前数据导出为 Excel，带时间戳，方便备份

## 🏗️ 技术选型

整个项目只用到了最基础的技术：

| 层级 | 技术 | 为什么这么选 |
|------|------|--------------|
| 前端 | 原生 HTML5 + CSS3 + Vanilla JS | 不需要框架，页面不复杂，原生够用 |
| 数据 | JSON（2648 条记录） | 体积小，加载快，直接 `fetch` 就能用 |
| 拼音 | `js/pinyin.js`（CJK 全字符映射） | 自己维护的映射表，覆盖中文、日文、韩文 |
| Excel 处理 | SheetJS（`xlsx.full.min.js`） | 业界标准的浏览器端 Excel 处理库 |
| PWA | `manifest.json` + `service-worker.js` | 实现桌面安装和离线缓存 |

整个项目就两个核心 JS 文件：`app.js`（搜索、渲染、拨号）+ `db.js`（数据存储层）。没有 Webpack，没有 Babel，没有 Node_modules，就是一个纯静态网站。

## 🐛 踩过的坑

### 坑1：Service Worker 必须在 HTTP 服务器下运行

**现象**：双击 `index.html` 打开，控制台报错"Service Worker registration failed"

**原因**：`file://` 协议下 Service Worker 无法注册，必须在 HTTP 服务器下运行

**解决**：本地用 `python -m http.server 8080` 或 `npx serve .` 启动，线上部署到 GitHub Pages，问题解决

### 坑2：办公电话区号补全逻辑

**现象**：有些号码存的是短号（如 `68560`），有些是完整号码，有些缺区号

**解决**：写了一个判断逻辑，11 位手机号不处理，5-8 位号码自动补 `0872-` 区号，短号不处理

### 坑3：拼音首字母映射表不全

**现象**：生僻字搜不到，比如"峤"、"茜"等非常用字

**解决**：扩展了 `pinyin.js` 的 CJK 字符映射表，覆盖 Unicode 中全部中日韩统一表意文字，生僻字也能正确匹配首字母

### 坑4：iOS Safari 对 PWA 的支持有差异

**现象**：安卓手机"添加到主屏幕"正常，iPhone 上找不到这个选项

**解决**：iOS 的入口在"分享"按钮里，叫"添加到主屏幕"，不是菜单里。在 `manifest.json` 里加了 `prefer_related_applications` 配置，提升 iOS 的 PWA 体验

## 💡 写在最后

这个项目最大的感受是：**做一个"能用"的内部工具，比想象中简单得多**。

不需要学最新的框架，不需要上云，不需要买服务器。一个静态网站 + 一点 JavaScript + PWA 技术，就能做出体验接近原生 App 的工具。

对我们医院来说，这个通讯录解决了 2648 条号码的查询问题。对其他单位来说，思路是一样的——把常用的数据做成一个 PWA，放到手机桌面上，大家都方便。

技术不一定要高大上，解决问题才是硬道理。
