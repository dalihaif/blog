## 📖 写在前面

在医院档案室工作这些年，我发现一个奇怪的现象——档案管理越来越规范，但**印章管理**却始终停留在"一个铁柜子 + 一本手写台账"的阶段。

每次有人来盖章，值班老师翻台账、找钥匙、开柜子、盖章、签字、锁柜子……一套流程下来少说五六分钟。要是赶上月底集中用章，排队能从办公室排到走廊。

更头疼的是**借出外带**——印章出了医院大门，什么时候回来？谁在用？用在什么文件上？全靠打电话问。万一丢了，翻遍台账也查不出个所以然。

所以我想：能不能像管理档案一样，给印章也建一个**全生命周期的数字化管理系统**？

经过几周的摸索和迭代，一套基于 **Python Flask + SQLite** 的印章管理系统诞生了。今天把开发思路分享出来，希望对同行有参考价值。

---

## 🎯 系统要解决什么问题？

先说痛点，再说方案。医院印章管理常见的几个老大难：

| 痛点 | 具体表现 |
|------|----------|
| **台账混乱** | 手写登记字迹潦草，查询全靠翻本子 |
| **借出失控** | 印章借出后无人跟踪，超期不还 |
| **刻制无序** | 新刻印章没有审批流程，想刻就刻 |
| **报废不明** | 废旧印章堆在柜子里，不知道哪些该销毁 |
| **统计困难** | 年终汇报全靠人工数，哪枚章用了多少次说不清 |

一句话总结：**印章从"出生"到"退休"，没人管全程。**

---

## 🏗️ 系统长什么样？

整套系统就两个核心文件 + 一个启动脚本：

```
印章管理系统/
├── server.py       ← Flask 后端（API + 数据库）
├── index.html      ← 主管理页面（11个功能模块）
├── usage.html      ← 移动端扫码登记页
└── start.bat       ← 一键启动
```

没错，**就这么多**。没有复杂的前后端分离，没有 Docker 容器，没有云数据库。一个 Python 文件管后端，一个 HTML 文件管前端，SQLite 存数据，轻得不能再轻。

### 11个功能模块一览

打开 `index.html`，顶部 Tab 栏一目了然：

1. **目录管理** — 所有印章的"户口本"
2. **刻制申请** — 新刻印章走审批流程
3. **使用登记** — 盖章/借出扫码登记
4. **借用管理** — 借出归还全跟踪
5. **回收检查** — 定期盘点印章去向
6. **报废销毁** — 废旧印章规范处置
7. **归档管理** — 印章退役后的档案化
8. **标签打印** — 一键生成印章标签
9. **统计报表** — 用章次数、借出频次一目了然
10. **系统设置** — 部门、审批人、印章类型配置
11. **操作日志** — 每一步操作都有迹可查

## 🔧 技术选型：为什么是 Flask + SQLite？

你可能要问：现在不都流行 Vue + Spring Boot + MySQL 吗？为什么选这么"简陋"的技术栈？

三个理由：

**1. 零成本部署**

医院内网环境，装个 Python 就能跑。不需要买服务器，不需要装数据库，不需要配 JDK。双击 `start.bat`，浏览器自动打开，完事。

**2. 零运维负担**

SQLite 是一个文件，备份就是复制文件。没有数据库密码，没有连接池配置，没有服务要守护。就算系统崩了，重装 Python、拷贝数据库文件，5分钟恢复。

**3. 够用就好**

医院印章数量通常在几十到几百枚之间，日使用记录几十条。SQLite 处理这个量级绰绰有余，响应速度毫秒级。杀鸡焉用牛刀？

## 💾 数据库设计：9张表管住印章一生

核心是9张表，每张表对应一个管理环节：

| 表名 | 作用 | 关键字段 |
|------|------|----------|
| `seals` | 印章档案目录 | 名称、类型、部门、保管人、状态 |
| `carvings` | 刻制申请 | 申请人、审批状态、刻制厂商 |
| `usages` | 使用登记 | 用印人、文件类型、盖章/借出、批次号 |
| `loans` | 借用管理 | 借出时间、预计归还、实际归还 |
| `recoveries` | 回收检查 | 盘点结果、异常标记 |
| `destructions` | 报废销毁 | 销毁方式、监销人、销毁日期 |
| `archives` | 归档记录 | 归档时间、归档位置 |
| `settings` | 系统配置 | 部门列表、审批人、文件类型 |
| `logs` | 操作日志 | 谁在什么时间做了什么 |

其中最精妙的设计是 **`usages`（使用登记）表**——它用 `usageType` 字段区分两种场景：

- **`use`（盖章使用）**：登记即完成，状态自动标记为"已完成"
- **`loan`（借出外带）**：登记后状态为"借出中"，需要手动归还

这样一张表就搞定了"用印"和"借出"两个业务场景，还支持**批量借出**——同一批次借出多枚印章，共享一个 `batchId`，归还时一键全部归还。

## 🚀 核心功能详解

### 1. 使用登记：扫码就能盖章

这是使用频率最高的功能。我们做了一个**移动端专用页面** `usage.html`，用手机扫码即可登记：

1. 扫描印章上的二维码 → 自动填入印章名称
2. 选择文件类型（证明/合同/报告等）
3. 填写用途和用印人
4. 提交，完成

整个流程不超过30秒。如果选择"借出外带"，还需要填写预计归还时间和联系电话，超期系统自动提醒。

### 2. 批量借出与归还

实际工作中经常遇到"一次借出3枚章去办事"的情况。系统支持：

- **批量借出**：勾选多枚印章，一次性登记，生成同一个批次号
- **批量归还**：点击"归还整批"，该批次所有印章同时标记为"已归还"
- **单枚归还**：也可以只归还其中一枚，其他继续借出

### 3. 刻制审批流程

新刻印章不是想刻就刻的，系统内置了审批流程：

> 提交申请 → 待审批 → 审批中 → 已批准 → 刻制中 → 已完成

每一步都有状态标记，审批人可以批准或驳回。如果驳回，还需要填写驳回原因。

### 4. Excel 导入导出

已有台账数据的单位，不用手工一条条录入。系统支持：

- **导入**：直接上传 Excel（.xlsx）或 CSV 文件，自动解析并入库
- **导出**：任意模块的数据都可以导出为 Excel，方便备份和上报

> ⚠️ 小贴士：导入 CSV 时注意编码问题，系统已自动处理 UTF-8 BOM 头，避免中文乱码。

### 5. 标签打印

每枚印章都可以生成标签，贴在存放盒上。标签包含：

- 印章名称和类型
- 保管部门
- 刻制日期和有效期
- 二维码（扫码直达使用登记页）

## 🐛 踩过的坑

开发过程中踩了不少坑，记录几个典型的：

### 坑1：端口冲突导致启动失败

**现象**：双击 `start.bat` 后页面显示"无法连接后端服务"

**原因**：上一次关闭不彻底，Python 进程还占着5100端口

**解决**：在启动脚本里加了自动杀旧进程的逻辑——启动前先检查5100端口，有旧进程就自动杀掉

### 坑2：Excel 导入中文乱码

**现象**：导入的 Excel 文件中，中文全部变成乱码

**原因**：浏览器的 `FileReader` 默认不按 UTF-8 解码

**解决**：读取时强制指定 `UTF-8` 编码，并自动去除 BOM 头

### 坑3：数据库字段不够用

**现象**：新需求要加字段，但不想影响已有数据

**解决**：用 `ALTER TABLE` + `try/except` 的迁移模式，启动时自动检查并添加新字段，已有数据完全不受影响

```python
migrations = [
    "ALTER TABLE usages ADD COLUMN usageType TEXT DEFAULT 'use'",
    "ALTER TABLE seals ADD COLUMN usageCount INTEGER DEFAULT 0",
]
for m in migrations:
    try: db.execute(m)
    except: pass  # 字段已存在则跳过
```

## 🔮 部署新单位：15分钟搞定

这套系统已经可以快速复制到其他单位。标准流程：

1. 拷贝整个 `印章管理系统/` 目录
2. 全局搜索替换旧单位名称 → 新单位名称
3. 修改系统设置里的部门和审批人
4. 删除旧数据库（让系统自建空库）
5. 双击 `start.bat` 启动
6. 通过导入功能批量导入印章 Excel 数据

全程大约15-30分钟，不需要写一行代码。
